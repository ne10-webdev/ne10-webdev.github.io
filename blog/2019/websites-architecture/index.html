Websites arhitechure's - SSE vs SPA vs Static

It was simple before. SSE ruled everything just 15 years ago, but not anymore. There's a bit of misunderstanding about what, when and why to use nowadays. I tried to dig for a reasons why SPA fills more complicated to develop than SSR websites. And what advantages/disadvantages each type has. Theres no one type that can be used universally yet.

Lets start from simplest

Static websites

Pros:

  • runs on any webserver
  • easy deploy/configuration

Cons:

  • slow deploy or complicated engine when it handles lots of content

Somehow it fills most people think if you do static website you must abandon any DB and use only plain text / templates files on filesystem as a data source for static engine. But why so? What makes static website is its engine that pre-generate web pages beforehand and than any webserver can serve them. We can have any data source connected to such engine, it can be files on filesystem, SQL / NoSQL DB, 3d party API, 3d party app that generate data, anything. It's all about how your users prefer to manage content.

If you choose for example markdown files on filesystem, than you don't need to build web based UI to manage content. But you also can have admin part as SPA with API connected to local SQL DB to manage content for static website. You'll have all advantages on admin part
and all advantages of static website on front user part. It can be monolithic on one server or distributed as few servers where remote admin part deploy static files to standalone user-facing webserver. Its best of both worlds and not over-complicated.

Server Side Rendering (SSR)

Pros:

  • straightforward development

Cons:

  • server load spent for rendering templates
  • can have slow UI

Used to be the only choose and kinda default choose today. Or probably should be so. I tried to figure out for myself why it fills like programming SSR is easier than SPA. Until i tried to make same code work on both sides i have not seen easy answer. It felt like i deal with less complexity on SSR beacause time span when my code are running for me fills as short and limited - from HTTP request to response. While on SPA my code running all the time when page are open and data in local state changes many times from different events.

Data flow on SSR most of the time are easy, you get params from user http request, call DB, prepare data for template, serve results back with response. Data flow always in same direction with mostly similar number of steps/events. On client side data mostly don't have state In old days of jQuery-like architecture, data had life only in DOM.

Single Page Application (SPA)

Pros:

  • fast UI when business logic can be processed locally
  • less compute demands on backend side

Cons:

  • huge size footprint of JS frameworks: load speed, cpu usage, etc.
  • complicated logic
  • not easy to parse such webapps for 3d-party apps (search engines, indexers, content parsers, etc)

It's most complicated architecture for sure. JS bloat on client side leads to huge disadvantages. When it is used where it's avoidable it's slows everything. HTML pages not that huge in size after compression compared to JSON responses. If end user only shortly interact and leave after visiting few pages than you probably don't want SPA. Not only load times are bigger, but more CPU and GPU usage leads to laggy experience, specially on low powered devices. Consider amount of JS that client has to compile and execute, amount of page content changes that leads to re-render's. Web was build for simple texts but became platform for everything today. There's almost no point to make website with mostly article-like content to be SPA, but many do.

Data flow more complicated than on SSR. Actually you have to be aware of two data flows - on SPA client side and back-end. Twice the work. And data flow on SPA can have many direction. Same component will get events with data from user and will communicate with API in both direction. To deal with data it will have its own state.

SSR can handle dynamic live data via jQuery-like code that update only parts of page. Its ok as for me, not fancy, but it works. Partly applying SPA to big SSR app also possible.

SPA development became over complicated fast. And mostly without a good reason. Not every project are on the same scale of complexity so there no golden tool. SPA frameworks add new problems we haven't had before and require time that can be spent elsewhere.
